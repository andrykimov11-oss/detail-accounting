# Техническое задание для программиста 1С

**Кому:** программист 1С
**Дата:** 2026-07-22
**Приоритет:** высокий — на критическом пути пилота
**Связанные документы:** `docs/1c-export-spec.md`, `docs/status-writeback-spec.md`

---

## 1. Общая картина

Разрабатывается внешнее приложение подетального учёта производства. Оно
считает, какие детали прошли операции цеха, на основе сканирования бирок.
1С остаётся источником истины для финансов, справочников и статуса заказа.

От 1С требуются **две доработки**, обе небольшие:

1. **Добавить поле `operation_id`** в выгрузку плана производства.
2. **Сделать HTTP-сервис** приёма статуса операции от внешнего приложения.

Перестраивать учёт в 1С не нужно. Это единственные изменения на стороне 1С.

```
1С ──[ выгрузка плана + operation_id ]──►  внешнее приложение
                                              (считает факт по сканам)
1С ◄──[ HTTP: статус операции ]────────────  внешнее приложение
        (устанавливает статус)                (когда все детали отсканированы)
```

---

## 2. Задача 1: поле `operation_id` в выгрузке плана

### 2.1. Что есть сейчас

1С выгружает план производства (отчёт «производственные операции»). Одна
строка = одна операция цеха по заказу. Внешнее приложение эту выгрузку
читает.

### 2.2. Что добавить

Добавить в выгрузку колонку **`operation_id`** — уникальный и **стабильный**
идентификатор строки операции в 1С.

| Требование | Пояснение |
|---|---|
| Уникальность | Разные операции — разные `operation_id` |
| Стабильность | Один и тот же `operation_id` при повторных выгрузках той же операции |
| Формат | Строка. GUID строки документа 1С предпочтителен |

Пример: `op_6564_3`, либо GUID регистра/строки табличной части — что
надёжнее в вашей конфигурации.

### 2.3. Зачем это нужно

Когда внешнее приложение посчитает, что операция выполнена (все детали
отсканированы), оно отправит в 1С именно `operation_id`, чтобы 1С нашла
нужную строку операции и установила статус. Без стабильного идентификатора
связка «внешнее приложение → 1С» невозможна.

### 2.4. Если `operation_id` получить сложно

Допустима альтернатива на время пилота: ключ из пары
`(номер заказа + имя операции)`. Менее надёжно (имя операции может
измениться в справочнике), но для пилота приемлемо. **Предпочтителен всё же
`operation_id`.**

### 2.5. Полная структура выгрузки

Колонки выгрузки плана (подробно в `docs/1c-export-spec.md`):

| # | Колонка | Тип | Обяз. | Пример |
|---|---|---|---|---|
| 1 | `order_num` | число | да | `6564` |
| 2 | `order_full_num` | строка | да | `ЛД00-006564` |
| 3 | `client_name` | строка | да | `Спецторг ООО` |
| 4 | `order_status` | строка | да | `К выполнению` |
| 5 | `deadline` | дата | да | `2026-06-24` |
| 6 | **`operation_id`** | строка | **да — NEW** | `op_6564_3` |
| 7 | `operation_name` | строка | да | `Раскрой плиты 16мм` |

**Важно про номер заказа.** Внешнее приложение уже умеет разбирать полный
номер (`ЛД00-006564`) на префикс и число. **Просьба оставлять префикс** —
он нужен для однозначной связки заказа. Именно потеря префикса при переносе
в Базис была корневой проблемой проекта, и мы её решили за счёт того, что в
выгрузке 1С префикс есть.

### 2.6. Что НЕ выгружать

Эти поля больше не нужны — статус и исполнителя теперь считает внешнее
приложение из сканера:

- статус операции (Выполнена/Передана);
- исполнитель;
- дата выполнения;
- телефон клиента (не нужен цеху, риск утечки ПД по ФЗ-152).

## 3. Задача 2: HTTP-сервис приёма статуса

### 3.1. Что сделать

HTTP-сервис в 1С, принимающий POST-запрос со статусом операции. По
`operation_id` находит строку операции в документе и устанавливает статус
«Выполнено».

### 3.2. Формат запроса

**Метод:** `POST`
**Content-Type:** `application/json; charset=utf-8`

**Тело запроса:**

```json
{
  "operation_id": "op_6564_3",
  "order_num": 6564,
  "operation": "Раскрой плиты 16мм",
  "status": "Выполнено",
  "completed_at": "2026-07-22T10:15:00",
  "scanned_details": 295,
  "planned_details": 295
}
```

| Поле | Назначение |
|---|---|
| `operation_id` | ключ поиска строки операции (из Задачи 1) |
| `order_num` | номер заказа (для проверки/логирования) |
| `operation` | имя операции (для проверки/логирования) |
| `status` | всегда `Выполнено` |
| `completed_at` | момент закрытия операции, ISO 8601 |
| `scanned_details` | сколько деталей отсканировано |
| `planned_details` | сколько планировалось (равно scanned при закрытии) |

### 3.3. Что должен делать сервис

1. Найти строку операции по `operation_id`.
2. Установить статус операции «Выполнено».
3. Вернуть HTTP-ответ:
   - **200** — статус успешно установлен;
   - **404** — операция с таким `operation_id` не найдена;
   - **4xx/5xx** — иная ошибка, с текстом в теле ответа.

### 3.4. Идемпотентность

Один и тот же статус может прийти повторно (например, после переотправки
из очереди). Повторный запрос по уже закрытой операции должен вернуть
**200** и ничего не ломать — не создавать дублей, не менять данные. Внешнее
приложение считает операцию закрытой при ответе 200.

### 3.5. Авторизация

Способ авторизации endpoint — на ваше усмотрение (Bearer-токен, Basic-auth,
служебная учётная запись 1С). Внешнее приложение поддерживает передачу
токена в заголовке `Authorization: Bearer <token>`. **Просьба сообщить**
выбранный способ и выдать тестовый доступ.

### 3.6. Что нужно от вас для подключения

1. URL endpoint (тестовый и боевой).
2. Способ авторизации и тестовый токен/учётную запись.
3. Точное имя реквизита статуса операции, если оно важно для отладки.

До готовности сервиса внешнее приложение работает в режиме заглушки:
считает статус и хранит его в очереди, но в 1С не отправляет. Как только
дадите endpoint — переключение на боевой режим на нашей стороне занимает
минуты и не требует изменений в вашем коде.

---

## 4. Порядок и сроки

Обе задачи независимы и могут делаться параллельно. Задача 1 (поле
`operation_id`) — проще и нужна раньше: без неё внешнее приложение не сможет
адресовать статус. Задача 2 (HTTP-сервис) может быть готова позже, к началу
пилота.

Обе задачи — на критическом пути пилота: пока их нет, подетальный учёт
работает, но результат в 1С не попадает. Просьба запланировать работы
заранее.

---

## 5. Что мы обеспечиваем со своей стороны

- Формирование корректного JSON по формату выше.
- Очередь с гарантией доставки: если сервис недоступен, статус не теряется
  и переотправляется автоматически.
- Дедупликацию: статус одной операции отправляется один раз (плюс возможные
  переотправки при сбоях — отсюда требование идемпотентности).
- Проверку связки заказов на нашей стороне (префикс + клиент + срок).
