**Область:** пилот на кромлении

## 1. Монтаж приложения в локальной сети

### 1.1. Что где стоит

```
┌──────────────────────────────────────────────┐
│  СЕРВЕР — один компьютер в цехе                │
│                                                │
│  • программа (Python + Flask)                  │
│  • база prod.db  ← единственное место факта    │
│  • общая сетевая папка с .xbir и выгрузкой 1С  │
│                                                │
│  IP в сети цеха: например 192.168.1.50         │
│  порт: 5001                                    │
└───────────────────────┬────────────────────────┘
                        │  Wi-Fi цеха
          ┌─────────────┼─────────────┐
        📱             📱             📱
     телефон 1      телефон 2      телефон N
     (браузер,      (браузер,      (браузер,
      камера)        камера)        камера)
```

### 1.2. Требования к серверу

| Что | Значение |
|---|---|
| ОС | Windows / Linux / macOS — любая с Python 3.10+ |
| Мощность | обычный офисный ПК, ничего специального |
| Сеть | проводом или Wi-Fi в той же сети, что и телефоны |
| IP | желательно **статический** (или закреплённый в роутере по MAC), чтобы адрес не менялся |
| Порт | 5001 открыт в брандмауэре для входящих в локальной сети |
| Пакеты | `pip install -r requirements.txt` |

### 1.3. Пошаговый монтаж

**Шаг 1. Поставить программу на сервер.**
```bash
# скопировать репозиторий на сервер, затем:
cd detail-accounting
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

**Шаг 2. Закрепить IP.**
В настройках сети сервера задать статический адрес (или в роутере
привязать IP к MAC сервера). Записать этот адрес — по нему будут заходить
телефоны.

**Шаг 3. Открыть порт 5001** во входящих правилах брандмауэра сервера
(только для локальной сети).

**Шаг 4. Проверить сеть.** С телефона открыть в браузере
`https://<IP-сервера>:5001/` — должна открыться страница входа.

**Шаг 5. Автозапуск (опционально).** Чтобы не запускать вручную каждое утро
— оформить как службу (systemd на Linux, «Планировщик заданий» на Windows).
Для пилота можно запускать и вручную командой из раздела 3.

### 1.4. Адреса

| Кому | Адрес |
|---|---|
| Оператор (телефон) | `https://<IP-сервера>:5001/` |
| Технолог (админка операторов) | `https://<IP-сервера>:5001/admin` |

Подробности про HTTPS и сертификат — `docs/Развёртывание_пилот.md`.

## 2. Откуда берётся детализация заказа

Ключевой момент: на кромлении **детализация заказа (какие детали) и факт их
прохождения — из разных источников.**

### 2.1. ПЛАН — какие детали в заказе

| Источник | Что даёт | Формат | Кто создаёт |
|---|---|---|---|
| **Базис-раскрой** | состав деталей заказа: размеры, кромка, количество, GUID | `.xbir` (XML) | технолог/раскройщик выгружает из Базиса |
| **Выгрузка 1С** | номер заказа, клиент, срок, список операций цеха | `.xlsx` | 1С по расписанию или по кнопке |

Это **обязательные** источники для пилота. `.xbir` даёт детали, выгрузка 1С —
привязку заказа к клиенту и операциям. Их связка (`link`) и импорт (`import`)
— подготовка базы перед сменой.

### 2.2. ФАКТ — какие детали реально прошли операцию

| Источник | Что даёт | Как |
|---|---|---|
| **Сканирование камерой** | факт: деталь прошла кромление | оператор сканирует бирку телефоном |

**На кромлении факт берётся ТОЛЬКО из сканирования.** Логи станка здесь не
участвуют.

### 2.2а. Путь к логам станков — заглушка, на этапе логов станет списком

В админке пока **одно** поле «путь к логам станков». Это **заглушка на
будущее**, а не рабочий механизм: в пилоте кромления логи не используются.

Логи физически лежат **каждый на своём станке** (`\\станок1\логи`,
`\\станок2\логи`, …). Когда дойдём до их сбора (этап после пилота, на
раскрое), поле превратится в **список путей по станкам** — таблицу
«станок → путь к его логам» в админке. Приложение будет обходить эти пути
**само**, читая каждый станок напрямую.

**Сторонняя программа-сборщик логов в одну папку не нужна** — это лишнее
звено, которое ломается. Приложение читает станки напрямую, как уже читает
папку Базиса. Механизм не заложен сейчас намеренно: формат привязки лога к
детали выяснится на том этапе, под него и будет сделано.

### 2.3. Про логи станков — почему их нет в пилоте кромления

В цехе есть логи станков (Sigma, Gabbiani — распил; Nanxing — присадка). Они
относятся к **другим участкам**, не к кромлению:

- **Логи распила** (Sigma/Gabbiani) — потенциальный автоисточник факта для
  участка **раскроя**, не кромления. У кромочного станка такого лога,
  привязываемого к детали, нет — поэтому факт кромления берётся сканом.
- **Nanxing `.SCX`** — данные сверления для участка **присадки**. Дают
  нормирование, но GUID детали в них нет (сквозной учёт присадки пока
  невозможен). К кромлению отношения не имеют.

Вывод: **для пилота на кромлении логи станков не нужны.** Они войдут в игру
на Фазе F, когда учёт расширится на раскрой (там встанет вопрос
автоотметки из логов) и присадку.

## 3. Как данные попадают в приложение

Прямой ответ на вопрос «приложение парсит папки или логи надо переносить
сторонними программами»:

### 3.1. Приложение читает файлы НАПРЯМУЮ из указанной папки

**Переносить файлы сторонними программами НЕ нужно.** Команды импорта
принимают как отдельные файлы, так и **папку целиком** — папка обходится
рекурсивно. Достаточно указать общую сетевую папку, куда Базис складывает
раскрои:

```bash
# импорт всех .xbir из папки Базиса (читает на месте, ничего не перемещает)
python3 main.py --db prod.db import "/путь/к/папке/выгрузка Базис" --require-link
```

Проверено: команда прочитала 145 файлов `.xbir` прямо из папки, 2611 деталей,
68 заказов — без какого-либо переноса.

### 3.2. Рекомендуемая схема потока данных

```
Базис  ──выгружает .xbir──►  ┌──────────────────────┐
                            │  ОБЩАЯ СЕТЕВАЯ ПАПКА  │
1С     ──выгружает .xlsx──►  │  на сервере          │
                            └──────────┬───────────┘
                                       │  приложение читает
                                       │  отсюда напрямую
                                       ▼
                            команда link + import → prod.db
```

Базис и 1С кладут свежие выгрузки в **одну расшаренную папку на сервере**.
Приложение читает их оттуда командой. Никаких промежуточных «переносчиков».

### 3.3. Что запускается и когда

Импорт — **не непрерывный**, он запускается командой (осознанное решение для
пилота: предсказуемо и просто). Порядок подготовки перед сменой:

```bash
# 1. свежая выгрузка 1С разложена в общую папку, .xbir тоже
# 2. разрешить связки заказ↔1С
python3 main.py --db prod.db link "/общая/папка/выгрузка Базис" \
    --plan "/общая/папка/производственные операции.xlsx"

# 3. разобрать очередь (если что-то не связалось)
python3 main.py --db prod.db pending

# 4. импортировать связанные заказы
python3 main.py --db prod.db import "/общая/папка/выгрузка Базис" --require-link
```

Это делает технолог утром (или когда пришли новые заказы). После этого
операторы сканируют — факт пишется в `prod.db` в реальном времени.

### 3.4. Если нужен автоматический импорт

Сейчас автослежения за папкой нет. Если по итогам пилота захочется, чтобы
приложение само подхватывало новые `.xbir` без ручной команды — это
добавляется одним из двух способов:

- **Планировщик** (cron / «Планировщик заданий»): запускать `link`+`import`
  каждые N минут или в начале смены. Проще всего, не требует нового кода.
- **Слежение за папкой** (watcher): импорт срабатывает при появлении файла.
  Требует небольшой доработки — делается после пилота, если понадобится.

Для пилота рекомендуется ручной запуск: технолог контролирует, что и когда
попадает в учёт, а очередь ручного разбора связок (`pending`) всё равно
требует его внимания.

## 4. Итог: минимальный чек-лист данных перед пилотом

- [ ] Сервер собран, IP закреплён, порт 5001 открыт, страница входа
      открывается с телефона.
- [ ] Общая сетевая папка на сервере создана; Базис и 1С кладут выгрузки туда.
- [ ] Свежая выгрузка 1С («производственные операции») в папке.
- [ ] `.xbir` пилотных заказов в папке.
- [ ] Связки разрешены (`link`), очередь `pending` разобрана.
- [ ] Заказы импортированы (`import --require-link`).
- [ ] Операторы заведены (админка `/admin` или `import-operators`).
- [ ] Справочник операций загружен (`rules`).
- [ ] Бирки с новым QR напечатаны (ТЗ_бирка_Базис.md).

Пункты про бирки и связь с 1С — на критическом пути: без них учёт считается,
но результат в 1С не уходит, а сканировать нечего.
