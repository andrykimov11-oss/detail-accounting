# Запись статуса операции в 1С (Этап B3)

**Версия:** 1.0
**Дата:** 2026-07-22
**Статус:** реализовано (заглушка-лог), боевой HTTP — по готовности сервиса 1С
**Код:** `src/one_c_writer.py`

## Что делает

Когда все плановые детали операции отсканированы, статус «Выполнено»
отправляется обратно в 1С. Это замыкает контур данных: 1С даёт план,
приложение считает факт и возвращает статус (docs/architecture.md, решение 4).

## Гарантия доставки

Статус не отправляется «в никуда». Схема:

```
операция закрыта → очередь (status_outbox) → транспорт → 1С
                                                 │
                        delivered=1 только при успешном ответе
```

Если 1С недоступен, запись остаётся в очереди со `delivered=0` и уходит при
следующем вызове `flush`. Статус операции не теряется, даже если сервис 1С
лежал в момент закрытия.

Дедупликация по `operation_id`: повторное закрытие той же операции обновляет
запись в очереди, а не создаёт дубль. Операция закрывается один раз.

## Заглушка на время пилота

HTTP-сервис 1С разрабатывает программист 1С, и на момент пилота он может быть
не готов: поле `operation_id`, endpoint и авторизация ещё уточняются
(docs/1c-export-spec.md). До этого writer работает в режиме `LogTransport` —
пишет payload в очередь и лог, но никуда не отправляет.

Переключение на боевой режим — замена транспорта на `HttpTransport`,
вызывающий код не меняется:

```python
from one_c_writer import HttpTransport
core.push_status_to_1c(ctx, ops, transport=HttpTransport(endpoint, token))
```

## Формат payload

```json
{
  "operation_id": "op_6564_3",
  "order_num": 6564,
  "operation": "Раскрой плиты 16мм",
  "status": "Выполнено",
  "completed_at": "2026-07-22T10:15:00",
  "scanned_details": 295,
  "planned_details": 295
}
```

## Применение

```bash
# Отправить статусы закрытых операций заказа
python main.py --db prod.db push --order 6564 \
    --ops "Раскрой плиты 16мм" "Облицовывание кромки 19/0,8"

# Повторить недоставленное (например, по расписанию)
python main.py --db prod.db flush
```

## Внешняя зависимость

Поле `operation_id` в выгрузке 1С и сам HTTP-сервис приёма — задача
программиста 1С, вне этого кода. На критическом пути пилота: без сервиса
статус считается и хранится, но в 1С не попадает. Запускать разработку
сервиса нужно параллельно.
