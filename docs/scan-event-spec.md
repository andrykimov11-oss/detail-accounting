# Спецификация события сканера (факт по детали)

**Версия:** 0.1
**Дата:** 2026-07-09
**Статус:** проект (для согласования с технологом)

## Назначение

Определяет, что именно фиксируется при одном сканировании бирки детали
оператором у станка, и как это событие превращается в факт прохождения
деталью операции. Это «входной язык» агента №3 (сбор факта).

Контекст: docs/architecture.md, решение 2 (единая точка ввода факта — сканер)
и решение 3 (статус операции выводится из факта по деталям).

## Принцип 1: участок → набор операций (оператор выбирает)

Оператор **не вводит операцию руками**, но и не привязан к одному станку.
Поток такой:

1. Оператор приступает к смене и **регистрируется на участке** (логин/бейдж).
2. За участком закреплён **набор операций** (справочник участок→операции).
3. Оператор сканирует QR детали → видит **только операции своего участка**,
   применимые к этой детали, и **выбирает нужную из списка**.
4. Никакого ручного ввода названия операции — только выбор из фильтр. списка.

**Справочник «участок → операции»** (см. также `operation_mapping.py`):

| Участок | Операции 1С (закреплены) |
|---|---|
| Раскрой | Раскрой плиты 10мм, 16мм, 16мм (свыше 30 м.п.) |
| Кромление | Облицовывание кромки 0,8 / 2 / 0,4 |
| Сверление | (источник присадки — отдельный, см. архитектуру) |
| Упаковка | Упаковка (позаказная, не через сканер) |

Один станок может быть универсальным (режет и 10, и 16) — оператор видит
обе операции и выбирает по факту обрабатываемого материала.

## Принцип 2: формат события сканера

При одном «бип» сканер присылает в систему **одно событие**:

```json
{
  "scan_id": "evt_20260709_101523_a3f1",
  "qr_code": "3c39598e6a",
  "area_id": "area_edging",
  "operator_id": "op_mazein",
  "operation_1c": "Облицовывание кромки 19/0,8",
  "scanned_at": "2026-07-09T10:15:23+05:00"
}
```

Поля:
- `scan_id` — уникальный ID события (генерирует система, для идемпотентности).
- `qr_code` — содержимое QR бирки, `MD5(GUID)[:10]` (см. qr-format-spec.md).
- `area_id` — ID участка, где оператор зарегистрировался в начале смены.
- `operator_id` — ID оператора (по бейджу/логину при регистрации на участке).
- `operation_1c` — выбранная операция (из списка операций участка, отфильтрованного по детали).
- `scanned_at` — время сканирования, ISO 8601 с таймзоной.

**Оператор не вводит название операции текстом.** Список операций формируется
системой как пересечение: (операции участка) ∩ (операции, применимые к детали
по её характеристикам). Оператор только выбирает из этого списка.

## Принцип 3: счётчик деталей по GUID (без экземпляров)

В `.xbir` деталь может иметь `qty=3` — это 3 одинаковые физические детали
с одним GUID и одним QR. Для учёта важно **общее количество** и
**идентификация конкретной потерянной детали**, а не номер экземпляра.

**Решение: счётчик отсканированных деталей по GUID.**

Оператор сканирует один и тот же QR столько раз, сколько физических деталей
обработал. Система ведёт счётчик `scanned_count` по GUID в рамках операции:

```
деталь GUID=X, qty=3 (из .xbir), 350x144x16мм, поз.5
  скан 1: scanned_count=1
  скан 2: scanned_count=2
  скан 3: scanned_count=3  →  плановое qty достигнуто
  скан 4: scanned_count=4  →  ПРЕВЫШЕНИЕ плана (нужно 3, отсканировано 4)
```

Потерянная/бракованная деталь идентифицируется GUID + характеристиками
(размер, позиция), а не номером экземпляра. В отчёте: «не отсканировано
1 шт детали 350x144x16мм поз.5 (GUID X)».

Альтернатива (если оператору неудобно сканировать 295 раз при раскрое):
на раскроечном участке — **групповое сканирование** (см. Принцип 5, режим B).

## Принцип 4: дубликаты и идемпотентность

**Двойное проведение сканером** (случайный повтор) — не должен удваивать факт.
Но в заказе **могут быть несколько одинаковых деталей** (тот же QR),
которые оператор legitimately сканирует подряд.

Правило: событие с тем же `(qr_code, operation_1c)` в течение **5 секунд**
после предыдущего — считается дубликатом и **отбрасывается** с предупреждением
оператору. Сверх 5 секунд — засчитывается как новая деталь (следующий
экземпляр того же QR).

5 секунд достаточно: за это время оператор физически не успеет взять и
обработать новую деталь. Порог вынесен в конфиг (`DUPLICATE_WINDOW_SEC = 5`).

## Принцип 5: два режима работы участка

Участки цеха различаются по характеру операций, и сканер работает по-разному.

**Режим A — поштучный (кромление, сверление).** Оператор обрабатывает
деталь за деталью, сканирует каждую. Классический «одно бип = одна деталь».

**Режим B — групповой (распил).** Станок режет карту раскроя — это
много деталей за один проход. Сканировать каждую нерационально
(45 уникальных × несколько плит). Здесь оператор сканирует **карту
раскроя** (один QR на партию деталей одной плиты), либо вводит
количество обработанных деталей на экране.

Режим B требует отдельного источника: связи «карта раскроя ↔ детали».
В `.xbir` есть `Номер плиты` и `Номер карты` — это основа для группового
факта. Спецификация группового режима — отдельный документ (после пилота
поштучного).

**Для пилота начинаем с режима A** (кромление) — там сканер даёт
максимальную ценность и наименее громоздок.

## Принцип 6: обработка ошибок

| Ситуация | Реакция системы |
|---|---|
| Неизвестный QR (нет в плане заказа) | Сообщение «деталь не найдена». Интерфейс показывает **список деталей заказа** — оператор выбирает нужную из списка. |
| Деталь не относится к операции (нет кромки, а скан на кромлении) | Интерфейс показывает **список деталей заказа**, применимых к операциям участка — оператор выбирает вручную. |
| Превышение плана (сканов больше qty) | Предупреждение: «превышение плана, проверьте». Факт сверх плана сохраняется отдельно (`overplan=true`), в закрытие операции не идёт. |
| Скан детали чужого заказа на участке | Ошибка: «другой заказ». |
| Повторный скан в течение 5 сек | Отбрасывается как дубликат (Принцип 4). |

**Список деталей при ошибке** — это fallback-режим ручного выбора. Показывает:
позицию, размер (Д×Ш×Т), материал, плановое qty. Оператор тапает по нужной —
факт засчитывается по выбранной детали. Это страховка на случай нечитаемого
QR, неверной разметки в `.xbir` или бирки не из этого заказа.

Все ошибки и предупреждения пишутся в лог событий (не теряются), а оператор
видит немедленную обратную связь на экране.

## Принцип 7: что получает агент расчёта статуса

После обработки событие сканера превращается в **факт по детали**:

```json
{
  "fact_id": "fact_20260709_101523_a3f1",
  "detail_uid": "EFA3C395-98E6-4E9E-A62B-3B80F0CF6298",
  "order_num": 6564,
  "operation_1c": "Облицовывание кромки 19/0,8",
  "instance_no": 2,
  "planned_qty": 3,
  "operator_id": "op_maslennikov",
  "workstation_id": "ws_edge_08_1",
  "completed_at": "2026-07-09T10:15:23+05:00",
  "anomaly": false
}
```

По этим фактам агент расчёта статуса операции сравнивает
`Σ instance_no` с `planned_qty` и закрывает операцию (см. архитектуру).

## Открытые вопросы (для пилота)

1. **Идентификация оператора.** Бейдж со штрихкодом? Логин в начале смены?
   RFID-браслет? — влияет на `operator_id`, решение на пилоте.

2. **Режим B (раскрой).** Групповое сканирование карты раскроя —
   спецификация после пилота режима A. На раскрое факта по детали не будет
   до реализации режима B.

3. **Оборудование сканера.** Какой сканер, как подключается к системе
   (USB-эмуляция клавиатуры / сетевой / планшет с камерой). Влияет на
   транспорт события (HTTP POST от планшета vs локальный ввод).

4. **Порог дубликата (30 сек).** Уточняется по поведению операторов на пилоте.

5. **Аномалия «деталь без кромки на кромлении».** Сколько таких наберётся
   за смену? Если много — проблема в разметке `.xbir`, не в сканере.
   Замеряется на пилоте.
