# Архитектура системы подетального учёта

**Версия:** 0.1
**Дата:** 2026-07-08
**Статус:** утверждённые решения куратора (на основе анализа реальных данных)

## Контекст проекта

Производство корпусной мебели из ЛДСП. Цех проходит технологический цикл:
распил → кромление → сверление → упаковка. Текущий учёт — в разрезе заказа
(в 1С), что не позволяет считать загрузку участков, отслеживать потерянные
детали и ставить точные сроки.

**Цель:** перейти на подетальный учёт, где первичный объект — деталь, а заказ —
её группировка.

## Зафиксированные архитектурные решения

### 1. Подетальный учёт — во внешнем приложении, не в 1С

1С остаётся источником истины для:
- финансов (оплата, смета, документы);
- справочников (клиенты, материалы);
- статуса заказа в целом.

Подетальный учёт развивается в **отдельном приложении**. 1С не перестраивается
под детали. Обоснование: проверка концепции подетального учёта в цехе — это
гипотеза; проверять её в рабочей 1С рискованно (встанут деньги). Внешнее
приложение позволяет пилотировать быстро и откатываемо. Если концепция
приживётся — перенос процессов в 1С на втором этапе.

### 2. Единая точка ввода факта — сканер

Оператор у станка сканирует **только один раз** (QR-код бирки детали).
Никакого дублирующего ручного ввода статуса операции в 1С. Это критично для
приживаемости процесса в цехе: два ввода на деталь «убивают» процесс.

### 3. Статус операции выводится из факта по деталям

Операция цеха (например, «Раскрой плиты 16мм») получает статус «Выполнено»
**автоматически**, когда все плановые детали этой операции отсканированы.
Не оператор ставит статус — система считает его по факту сканирования.

Правило закрытия операции:
```
операция «Выполнено»  ⇔  отсканировано = плановое количество деталей
```

Если плановое количество = 295, а отсканировано 294 → статус
«В работе, 1 деталь не отсканирована» (потенциальная потеря).

### 4. Запись статуса в 1С — через HTTP-сервис

Статус операции, посчитанный в нашем приложении, отправляется обратно в 1С
через **HTTP-сервис** (разрабатывается программистом 1С на стороне 1С).

Наше приложение → POST на endpoint 1С:
```
{
  "order_num": 6564,
  "operation": "Раскрой плиты 16мм",
  "status": "Выполнено",
  "completed_at": "2026-07-08T10:11:26",
  "scanned_details": 295,
  "planned_details": 295
}
```

1С-сервис устанавливает статус операции в документе заказа. Это единственная
доработка 1С в проекте (не перестройка учёта, а один сервис приёма статусов).

## Поток данных (итог)

```
       план                         факт
  .xbir (детали) ──►  наше приложение  ◄──── сканер (1 скан/деталь)
                          │              оператор у станка
  отчёт 1С ──────────────►│
   (состав заказа,        │
    клиент, срок)         ▼
                    расчёт статуса операции
                    «все детали отсканированы?»
                          │
                          ▼
                    HTTP POST → сервис 1С → 1С
                    (статус операции установлен)
```

## Источники данных и их роли

| Источник | Что даёт | Направление |
|---|---|---|
| `.xbir` (Базис-раскрой) | плановый состав деталей заказа | импорт в приложение |
| отчёт 1С (`.xlsx`) | состав заказа, клиент, срок, материалы | импорт в приложение |
| сканер (QR бирки) | факт прохождения деталью операции | ввод в приложение |
| HTTP-сервис 1С | приём посчитанного статуса операции | экспорт из приложения |

## Идентификация детали

- **Первичный ключ:** GUID из `.xbir` (`ID детали`).
- **QR-код бирки:** `MD5(GUID)[:10]` (см. `qr-format-spec.md`).
- **Связка заказ ↔ детали:** по нормализованному номеру заказа (числовой核心
  извлекается из `Номер заказа` в `.xbir` и из номера документа в 1С).
- **Связка операция ↔ деталь:** по характеристике, заданной в операции
  (толщина для распила, толщина кромки для кромления).

## Роли операций цеха (два уровня)

Анализ отчёта 1С показал, что операции делятся на два уровня:

**Уровень станка (физические операции над материалом):**
- Раскрой плиты 10мм / 16мм / 16мм свыше 30 м.п. — распил, по толщине ДСП
- Облицовывание кромки 0,8 / 2 / 0,4 — кромление, по толщине кромки
- Облицовывание криволинейных деталей, рез профиля, сдвойка — спецоперации

**Уровень документооборота (не над деталью):**
- Операция ОТК, приёмка на СГП, упаковка, передача клиенту

Для первого уровня применяется подетальный учёт (статус из сканера).
Для второго — статус остаётся позаказным (не требует сканера).

## Открытые вопросы

1. **Сверление/присадка.** В `.xbir` (Базис-раскрой) отверстий нет — это
   выгрузка программы оптимизации раскроя, а не модели изделия. Присадка
   задаётся в Базис-мебельщик и уходит в техдокументацию/программу
   присадочного станка. Нужен отдельный источник для нормирования участка
   сверления — в зоне агента №3 (сбор факта).

2. **Развёртка `qty > 1`.** Если `qty=3`, печатаются 3 бирки с одним QR.
   При сканировании надо различать экземпляры — возможно, суффикс `.1/.2/.3`.
   На этапе проектирования учёта.

3. **HTTP-сервис 1С: формат и авторизация.** Уточняется с 1С-ником:
   структура документа заказа, имена реквизитов операции, способ авторизации
   endpoint. До этого момента наше приложение пишет статус в свою БД, а
   запись в 1С — заглушка (лог).

4. **Что считать «плановым количеством» для операции.** Для распила 16мм —
   это 295 шт деталей. Для кромления 0,8 — это количество деталей с кромкой
   0,8 (а не погонных метров). Правила расчёта планового количества по каждой
   операции — отдельная спецификация (агент нормирования).
