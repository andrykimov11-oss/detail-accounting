# Связка «заказ Базис ↔ документ 1С»

**Версия:** 1.0
**Дата:** 2026-07-19
**Статус:** реализовано (Этап B1)
**Код:** `src/one_c_loader.py`, `src/order_resolver.py`

## Проблема

Номер заказа в 1С состоит из префикса и числа (`ЛД00-006564`). При переносе
в Базис префикс теряется — в `.xbir` остаётся только `6564`.

Проверка на массиве (238 файлов `.xbir`, 102 878 строк 1С) выявила **две**
причины неоднозначности, а не одну:

**1. Три префикса.** `ПС00` (5 476 заказов), `ЛД00` (3 296), `0Ч00` (1 814).
1 226 числовых номеров из 9 265 (13,2%) встречаются у нескольких префиксов.

**2. Номера переиспользуются по годам** — это выяснилось только при
реализации. Полный номер документа 1С **тоже не уникален**:

```
0Ч00-001991  20.11.2024  Иванов Игорь Сергеевич        [К выполнению]
0Ч00-001991  17.07.2025  Ларионов Алексей Александрович [К выполнению]
0Ч00-001991  09.06.2026  Беляев Александр Владимирович  [К выполнению]
```

798 номеров из 10 586 (7,5%) имеют несколько разных клиентов, и у **780 из
них несколько записей активны одновременно** — фильтрация по статусу не
помогает.

**Ключ заказа в 1С — это пара «полный номер + дата заказа»**, а не номер.
Группировка по одному номеру склеивает разные заказы и подставляет чужого
клиента.

## Решение: два независимых признака

**Признак 1 — срок исполнения.** Максимальный срок исполнения заказа —
**45 дней**. Заказ старше физически не может резаться сейчас. В 1С такие
записи просто не закрывают: из 11 354 записей «К выполнению» в окно
попадают 523, остальные 95% — незакрытые хвосты за два года.

**Признак 2 — имя клиента.** В `.xbir` поле «Номер заказа» содержит не
только число, но и клиента: `7936 Коновалов`. В 1С есть колонка «Клиент».

```
число из .xbir
   ├─ 1 кандидат в 1С               → UNIQUE
   └─ 2+ кандидатов
        ├─ сузить по сроку (45 дней)
        │     остался один          → RESOLVED_BY_WINDOW
        │     осталось 2+           → сравнить по клиенту
        │     не осталось (выгрузка → откат к полному списку,
        │        устарела)             сравнение по клиенту
        └─ клиент совпал с одним    → RESOLVED_BY_CLIENT
             иначе                  → MANUAL_REQUIRED, технологу
```

**Признаки проверяют друг друга.** Если срок указывает на один заказ, а имя
клиента — на другой, связка **не** устанавливается: расхождение означает,
что данные неверны, и решать должен человек.

### Опорная дата окна — дата выгрузки, а не сегодня

Выгрузка 1С делается не каждый день: на момент проверки массив был месячной
давности (свежайшая дата заказа 2026-06-22 при текущей 2026-07-22). Отсчёт
окна от сегодняшнего числа выбрасывал **10 живых заказов**, которые в момент
выгрузки были в работе. Поэтому опорная дата — самая свежая дата заказа в
самой выгрузке (`export_reference_date`).

### Откат при устаревшей выгрузке

Если все кандидаты старше окна, заказ **не теряется**: резолвер откатывается
к полному списку кандидатов и разбирает их по имени клиента. Окно только
сужает выбор, но никогда не отбрасывает заказ целиком.

### Принцип: отказ вместо угадывания

Резолвер **никогда не выбирает «наиболее вероятного» кандидата**. Если пара
не даёт однозначного ответа, заказ уходит технологу в очередь ручного
разбора. Заказ без подтверждённой связки не допускается к учёту: оператор
у станка физически не может отметить деталь по такому заказу.

Это исключает ошибку оператора и, что важнее, исключает *автоматическую
незаметную* запись статуса в чужой заказ.

### Сравнение наименований

Решение принимает **строгое пословное** сравнение (`names_match`), а не
оценка схожести:

```
«Коновалов»  ⊆ {коновалов, дмитрий, геннадьевич}  → совпало
«Спецторг»   == «Спецторг ООО»                     → совпало
«С групп»    ⊆ {компания, с, групп}                → совпало (дефис = разделитель)
«Петров»     ⊄ {кузнецов, петр, петрович}          → НЕ совпало
```

Нормализация убирает регистр, пунктуацию, дефисы, организационно-правовые
формы (ООО, ИП, ЗАО) и приводит `ё` к `е`.

**Почему не нечёткое сравнение.** Первая реализация сравнивала подстроки, и
«Петров» совпал с «Кузнецов **Петров**ич» со счётом 1.0 — заказ ушёл бы к
чужому клиенту автоматически. Русские фамилии слишком похожи
(Петров/Петрович, Иванов/Иванова), любой порог даёт ложные связки. Оценка
схожести сохранилась, но только **справочно** — она показывается технологу
в очереди, на решение не влияет.

**Защита от инициалов.** Совпадения только по коротким словам («м», «г»)
недостаточно: требуется хотя бы одно слово длиной от 3 символов.

## Результат на массиве

68 заказов раскроя против 11 354 активных заказов 1С:

| Итог | Заказов | Доля |
|---|---|---|
| Однозначно в 1С | 47 | 69,1% |
| По сроку исполнения | 1 | 1,5% |
| Разрешено клиентом | 20 | 29,4% |
| **Связка установлена** | **68** | **100%** |
| Технологу на разбор | 0 | 0% |

**Ручной разбор пуст.** Заказ `7846`, где в `.xbir` имени клиента нет вообще,
разрешается сроком исполнения: `ЛД00-007846` от 2025-07-04 вне окна, остаётся
`ПС00-007846` от 2026-06-16.

Выбор независимо подтверждается номенклатурой: материал `.xbir` «Орех Кария
16 мм» есть только у `ПС00-007846` («Орех Кария К 2750*1830*16мм Lamarty»),
у `ЛД00-007846` позиций нет вовсе. Два признака, полученные разными путями,
дают один ответ.

## Реестр связок

Разрешённая связка пишется в таблицу `order_links` и переиспользуется:

| Поле | Назначение |
|---|---|
| `order_num` | число из `.xbir` |
| `order_full_num` | документ 1С |
| `client_name` / `xbir_client` | клиент по данным 1С и Базиса |
| `status` | `unique` / `client` / `manual` / `manual_confirmed` / `not_found` |
| `reason` | чем разрешено — для аудита |
| `candidates` | кандидаты, среди которых выбирали |
| `confirmed_by` | кто подтвердил вручную |

Статус `manual_confirmed` отличается от автоматических: в аудите видно, где
решение принял человек.

## Работа технолога

```bash
# 1. Разрешить связки
python main.py --db prod.db link samples/**/*.xbir \
    --plan "производственные операции.xlsx"

# 2. Посмотреть, что не разрешилось
python main.py --db prod.db pending

# 3. Сопоставить вручную
python main.py --db prod.db confirm \
    --order 7846 --doc ПС00-007846 --by "Иванов И.И."

# 4. Импорт только связанных заказов
python main.py --db prod.db import samples/**/*.xbir --require-link
```

## Ограничения и что дальше

**Выборка.** Проверено на 68 заказах раскроя. В 1С 10 586 заказов — при
масштабировании доля ручного разбора может отличаться. Отчёт
`format_resolve_report` показывает разбивку по признакам, чтобы отслеживать
деградацию.

**Ручной ввод номеров технологом не требуется.** Он был бы источником
ошибок: заносить сотни номеров руками — гарантированные опечатки. Оба
признака берутся из данных, которые уже есть.

**Префиксы (`ПС00`, `ЛД00`, `0Ч00`) не трогаем** — они нужны другим учётным
системам. Задача решена без изменения нумерации.

**Материалы как третий признак.** Проверено вручную: выгрузка «Заказы
плитные с номенклатурой» разрешает и случай `7846` (материал «Орех Кария
16 мм» есть только у `ПС00-007846`). В код не вошло — оставлено на случай
роста очереди ручного разбора.
