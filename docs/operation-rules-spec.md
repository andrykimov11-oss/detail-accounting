**Версия:** 1.0

**Статус:** реализовано (Этап B2)
**Код:** `src/operation_rules.py`, `src/operation_plan.py`

## Как читаются характеристики

Названия операций 1С кодируют признаки отбора деталей:

```
«Раскрой плиты 16мм»              → детали толщиной 16 мм
«Рез ЛДСП 10, 16мм (по высоте)»   → детали 10 или 16 мм
«Облицовывание кромки 19/0,4»     → детали с кромкой 0,4 мм
«Облицовывание кромки 26,29/2мм»  → детали с кромкой 2 мм
```

Формат кромления — «толщина ДСП / толщина кромки».

**Толщина кромки проверена на данных.** На 20 заказах, где операция кромления
ровно одна, толщина кромки в деталях `.xbir` совпала с названием операции во
всех 20 случаях.

**Толщина ДСП из названия кромления не применяется.** В названиях стоят
номинальные значения (19, 26, 29, 35, 45), которых в `.xbir` не существует:
фактические толщины 3, 4, 10, 16, 18, 22 мм. Соответствие номинала факту —
вопрос к технологу, не к коду.

Толщины из названий раскроя фильтруются по реально встречающимся: «Раскрой
плиты 22мм и 26мм» даёт `(22,)`, потому что деталей 26 мм в `.xbir` нет.

## Тарифные надбавки

Решение куратора: **одна учётная операция на группу, остальные — надбавки.**

Часть операций — тарифные разновидности той же самой работы. Деталь
обрабатывается один раз, а строк в заказе несколько:

```
Раскрой плиты 16мм                    ← учётная, берёт план
Раскрой плиты 16мм (свыше 30 м.п.)    ← надбавка за объём
Раскрой плиты 16мм (ГЛЯНЕЦ)           ← надбавка за материал
Раскрой плиты 16мм (сборка)           ← та же работа в контуре сборки
```

Надбавки распознаются по маркерам в названии: `свыше`, `сложн`, `глянец`,
`матовый`, `сборка`, `черновой`, `нанесение клея`. Плана они не берут и
закрываются вместе с учётной операцией группы.

Из 227 операций: **49 учётных подетальных, 29 тарифных, 149 позаказных.**

**Исключение.** Если учётной операции в заказе нет, а надбавка есть, план
считает надбавка — иначе участок остался бы без плана вовсе. Реальный случай:
заказ 1991 содержит «Раскрой плиты 16мм (свыше 30 м.п.)» без базовой операции.

## Дедупликация плана внутри заказа

Без неё план раздувается, и **операция никогда не закроется**: отсканировано
всегда меньше планового. Это ломает главную функцию системы.

Замер на массиве до дедупликации: раскрой давал 22 834 детали при 10 615
физических — коэффициент 2,15.

Работают два правила.

**По предикату.** Если несколько операций отбирают одни и те же детали
(«Облицовывание кромки 19/2» и «35/2» — обе кромка 2 мм), план берёт одна.
Базовой считается операция с самым коротким названием: «Облицовывание кромки
19/2» короче, чем «Облицовывание криволинейных деталей 19/2». Это эвристика —
точное разделение требует признака от технолога.

**По участку.** На раскрое, фрезеровании и присадке деталь проходит участок
один раз, поэтому две операции одного участка не считают её повторно
(«Раскрой плиты 16мм» и «Рез ЛДСП 10, 16мм» пересекаются по деталям 16 мм).

На кромлении иначе: деталь с кромками 0,4 и 2 законно проходит **две**
операции. Там дедупликация идёт только по предикату.

Результат: раскрой 10 027 при 10 615 физических деталях, коэффициент 0,94.
Превышений нет ни в одном из 68 заказов.

## Что осталось нерешённым

**8 групп коллизий.** Операции с одинаковым предикатом, различающиеся тем,
чего нет в `.xbir`:

```
предикат «кромка 2 мм»:
    Облицовывание кромки 19/2
    Облицовывание криволинейных деталей 19/2
    Облицовывание узких деталей 19/2
    Облицовывание кромки 35/2
    …ещё 12
```

Признаки «криволинейная», «узкая», «косой рез» в выгрузке раскроя
отсутствуют. Сейчас работает эвристика «самое короткое название», и она
может отнести детали не к той операции внутри группы. На суммарную загрузку
участка это не влияет — деталь считается один раз, — но распределение между
операциями внутри участка приблизительное.

Точное разделение требует признака от технолога. Отчёт `main.py rules`
показывает все группы коллизий.

**122 операции без характеристики в названии.** «Еврозапил», «Сверление под
конфирмат», «Вырез под мойку» — признак отбора деталей из названия не
извлекается. Они помечены позаказными: лучше считать вручную, чем отобрать
не те детали. Участок присадки целиком в этой категории — ему нужен источник
Nanxing (Этап B3).

## Применение

```bash
# Построить правила и увидеть коллизии
python main.py --db prod.db rules --areas "Операции по участкам.xlsx"
```

В коде:

```python
from one_c_loader import load_area_operations
from operation_rules import build_rules, find_collisions
from operation_plan import build_order_plan

rules, report = build_rules(load_area_operations(path))
plans, warnings = build_order_plan(details, operations_1c, rules=rules)
```

Без параметра `rules` работает прежний встроенный набор из 12 правил —
режим отладки, обратная совместимость сохранена.
