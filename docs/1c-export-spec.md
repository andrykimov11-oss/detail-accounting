# Спецификация выгрузки из 1С: «План производства»

**Версия:** 0.1
**Дата:** 2026-07-09
**Назначение:** документ для программиста 1С / настройки отчёта

## Назначение

1С выгружает **плановый состав производства** для внешнего приложения
подетального учёта. Только план: какие заказы в работе, какие операции
по ним назначены. Никаких фактов выполнения — факт теперь считается во
внешнем приложении и пишется обратно через HTTP-сервис.

## Формат

**Один файл выгрузки:** `production_plan_YYYYMMDD.xlsx` (или .csv)
**Периодичность:** выгрузка по расписанию (1 раз в день утром) + по кнопке.
**Содержимое:** строки операций по **всем активным заказам** цеха.

## Структура колонок

| # | Колонка | Тип | Обязательное | Пример | Комментарий |
|---|---|---|---|---|---|
| 1 | `order_num` | число | да | `6564` | числовой номер заказа без префиксов и нулей |
| 2 | `order_full_num` | строка | да | `ЛД00-006564` | полный номер документа 1С |
| 3 | `client_name` | строка | да | `Спецторг ООО` | наименование клиента |
| 4 | `order_status` | строка | да | `К выполнению` | статус заказа: `К выполнению` / `В резерве` / `Закрыт` |
| 5 | `deadline` | дата | да | `2026-06-24` | плановая дата выдачи (ISO) |
| 6 | `operation_id` | строка | да | `op_6564_3` | **NEW**: уникальный ID операции в 1С, для обратной записи статуса |
| 7 | `operation_name` | строка | да | `Раскрой плиты 16мм` | имя операции цеха (как в справочнике) |
| 8 | `operation_stage` | строка | нет | `cutting` | укрупнённая стадия (опционально, можно не заполнять — приложение сопоставит) |

## Что НЕ выгружать (важно!)

| Колонка из старого отчёта | Почему убрать |
|---|---|
| `Статус` операции (Выполнена/Передана) | статус теперь считает внешнее приложение из сканера |
| `Исполнитель` | теперь из сканера (operator_id при регистрации на участке) |
| `Дата выполнения` | теперь из сканера |
| `Дата передачи в цех` | не используется |
| `Телефон клиента` | не нужен цеху, риск утечки ПД (ФЗ-152) |
| `Количество записей` | техническое поле 1С, не нужно |

## Фильтр строк

Выгружать только строки, где:
- `order_status` ∈ (`К выполнению`, `В резерве`) — **НЕ** выгружать `Закрыт`;
- операция относится к производству (не бухгалтерские/складские).

## Ключевое поле: `operation_id`

Это **новое поле**, которого нет в текущем отчёте. Оно нужно для обратной
записи статуса операции через HTTP-сервис (архитектура, решение 4).

Когда внешнее приложение посчитает, что операция выполнена (все детали
отсканированы), оно отправит в 1С:
```json
{
  "operation_id": "op_6564_3",
  "status": "Выполнено",
  "completed_at": "2026-07-09T10:15:00"
}
```

1С-сервис по `operation_id` находит строку операции и устанавливает статус.
Без этого ID связка «внешнее приложение → 1С» невозможна.

**Если operation_id получить сложно** — альтернатива: ключ
`(order_num + operation_name)`. Менее надёжно (имя операции может
измениться), но допустимо на pilots.

## Что даёт 1С vs что берёт

```
1С ──[ выгрузка плана ]──► внешнее приложение
                            (обрабатывает сканы,
                             считает статус операции)
1С ◄──[ HTTP: статус ]──── внешнее приложение
                            (когда все детали отсканированы)
```

## Два варианта реализации (для 1С-ника)

**Вариант 1 (простой, рекомендую для пилота):**
- Регламентное задание в 1С выгружает `.xlsx` в расшаренную папку
  по расписанию (например, в 7:00 каждый день).
- Имена колонок точно как в таблице выше.
- Внешнее приложение читает файл и импортирует.

**Вариант 2 (для промышленной эксплуатации):**
- HTTP-сервис в 1С с методом `GET /production_plan`.
- Внешнее приложение запрашивает план по сети.
- Плюс: актуальность в реальном времени, не нужно расшаривать папки.

Для пилота достаточно Варианта 1. Переход на Вариант 2 не требует
переписывания внешнего приложения — только замена источника данных
(файл → HTTP), делается в одном модуле.
