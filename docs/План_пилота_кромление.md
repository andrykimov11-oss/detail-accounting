# План пилота: подетальный учёт на участке кромления

**Дата:** 2026-07-22
**Участок:** Кромление (пилотный)
**Длительность:** 2 недели (10 рабочих смен)
**Статус:** к запуску после готовности бирок и связки с 1С

---

## 1. Зачем пилот

Ядро системы готово и проверено на данных: связка с 1С работает на 100%,
план не задваивается, статус уходит обратно в 1С, интерфейс оператора собран.
Но система ещё **ни разу не встречалась с живым цехом**.

Пилот отвечает на один вопрос: **приживается ли подетальный учёт в реальной
работе** — или ломается о поведение операторов, качество бирок, темп смены.

Пилот считается **успешным, даже если выявит проблемы**, при условии, что он
даст факты для решения. Провал — когда после двух недель непонятно, работает
подход или нет.

## 2. Почему именно кромление

- Правила отбора деталей покрывают основной объём участка и проверены на
  данных (толщина кромки совпала в 20 из 20 заказов).
- 77% деталей имеют `qty ≤ 2` — поштучный скан реалистичен, режим группового
  сканирования (как на раскрое) не нужен.
- Разрыв идентификации на присадке (Nanxing без GUID) сюда не влияет.

## 3. Границы пилота

**В периметре:**

- один участок — Кромление;
- поштучный скан бирки (режим A);
- 2–3 операции кромления (0,4 / 0,8 / 2 мм);
- вход операторов по имени без пароля, регистрация на участке.

**Вне периметра (не трогаем на пилоте):**

- остальные участки (раскрой, присадка, фрезерование, сборка);
- групповое сканирование карты раскроя;
- автоматические отметки из логов станка;
- запись статуса в боевую 1С — если сервис не готов, работает заглушка-лог,
  статус копится в очереди и отправляется позже.

## 4. Что должно быть готово до старта

Проверяется чек-листом готовности перед первой сменой:

1. **Бирки с новым QR** напечатаны для пилотных заказов (ТЗ_бирка_Базис.md).
   Без этого пилот не стартует.
2. **Выгрузка плана из 1С** получена, связка заказов разрешена
   (`main.py link`), очередь ручного разбора пуста или разобрана технологом.
3. **Заказы импортированы** в БД пилота (`main.py import --require-link`).
4. **Сканер** подключён к рабочему месту, печатает QR как текст + Enter.
5. **Рабочее место** у станка: экран, запущенный `operator_app.py`, операторы
   заведены в справочник.
6. **Операторы проинструктированы** (Инструкция_оператора.md).

## 5. Роли

| Роль | Кто | Что делает на пилоте |
|---|---|---|
| Оператор кромления | сменные операторы | сканирует бирки, работает с интерфейсом |
| Технолог | ответственный за учёт | готовит заказы, разбирает очередь связок, ведёт журнал |
| Куратор пилота | руководитель проекта | смотрит метрики, принимает решение по итогам |
| Поддержка | разработчик | чинит сбои, дорабатывает по обратной связи |

## 6. Метрики — что замеряем каждую смену

Заносятся в журнал пилота (форма ниже). Считаются из `scan_events` в БД плюс
наблюдение.

| Метрика | Как считаем | Целевой ориентир |
|---|---|---|
| Доля нечитаемых QR | сканов с ошибкой чтения / всего попыток | < 3% |
| Аномалий за смену | статусы overplan + wrong_operation + unknown_qr | < 5% сканов |
| Время на деталь | наблюдение / метки времени сканов | не растёт от смены к смене |
| Сходимость план/факт | операций закрыто корректно / всего операций | план сходится |
| Использование fallback | сколько раз оператор выбирал деталь из списка | снижается по мере привыкания |
| Отзывы операторов | короткий опрос в конце смены | нет саботажа, понятен процесс |

## 7. Критерии решения

**Пилот успешен → идём в Фазу F (расширение), если:**

- операторы **не саботируют** скан — не обходят систему, не копят бирки;
- **аномалий < 5%** за смену к концу первой недели;
- **план сходится**: операции закрываются, когда все детали отсканированы;
- **время на деталь приемлемо** — скан не тормозит работу у станка.

**Пилот выявил проблемы → чиним до расширения, если:**

- аномалий стабильно > 5% и причина в системе, а не в обучении;
- операторы обходят скан из-за неудобства интерфейса;
- план систематически не сходится (расхождение модели с реальностью).

**Пилот провалился → пересматриваем подход, если:**

- за две недели не удалось собрать данные для решения (не было бирок,
  сканера, заказов — организационный провал, не технический).

## 8. График

| Дни | Этап |
|---|---|
| −3…−1 | Подготовка: чек-лист готовности, обучение операторов |
| 1–2 | Запуск с поддержкой разработчика у станка, ловим очевидное |
| 3–5 | Самостоятельная работа операторов, ежедневный сбор метрик |
| 6–10 | Наблюдение, накопление статистики, разбор аномалий |
| +1 | Разбор итогов, решение по Фазе F |

## 9. Что делать при сбоях

- **Сканер не читает бирку** — оператор выбирает деталь из fallback-списка
  на экране (система показывает его автоматически). Скан не теряется.
- **Система зависла / экран не отвечает** — перезапустить `operator_app.py`.
  Факт не теряется: счётчики хранятся в БД и восстанавливаются.
- **1С недоступна** — статусы копятся в очереди, уходят позже через `flush`.
  На работу оператора не влияет.
- **Непонятная ошибка** — записать в журнал (заказ, деталь, что было на
  экране), продолжить работу, разработчик разберёт по логу `scan_events`.

## 10а. Что отдельно наблюдаем на пилоте

Два вопроса, по которым решение отложено до данных пилота:

**Деталь с несколькими кромками (0,4 и 2,0).** Сейчас оператор сканирует её
на каждой операции отдельно (переключил операцию → скан). Смотрим, удобно ли
это операторам и не путаются ли они. Если окажется громоздко — рассмотрим
вариант «один скан закрывает все кромки детали». Учёт это поддерживает, менять
только эргономику.

**Сбор логов станков** — на пилоте кромления не используется, но на следующем
этапе (раскрой) встанет вопрос автоотметки из логов. Логи лежат на разных
станках; приложение будет читать их напрямую списком путей, без сторонних
сборщиков. Формат привязки лога к детали проверяется на том этапе.

## 11. Результат Фазы E

На выходе — четыре вещи:

1. **Ответ «да/нет»** по критериям раздела 7.
2. **Замеренные метрики** за 10 смен (журнал + выгрузка из `scan_events`).
3. **Список доработок** из практики цеха.
4. **Обоснованное решение** о Фазе F: расширять на другие участки или сначала
   дорабатывать.
