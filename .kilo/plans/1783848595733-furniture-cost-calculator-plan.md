# План: Калькулятор материалов корпусной мебели (Streamlit)

## 1. Контекст и цель
Новое **отдельное** приложение (не часть `detail-accounting`) для оценки расхода
материалов на изделие корпусной мебели. На входе — цены (ЛДСП за м², кромка 0,4 мм
и 2 мм за п.м.), габариты изделия и выбор типа изделия из файла-справочника.
На выходе — общая площадь ЛДСП, длина кромки по толщинам и итоговая стоимость,
а также визуализация выбранного изделия.

**Рабочий корень проекта:** `/Users/andrey/Desktop/projects/конструктор мебели`
(там же лежит исходный `Конструктор_офисной_мебели_таблицы_v2.xlsx`).

## 2. Ключевые решения
- **Стек:** Python 3.11+, Streamlit (веб-UI), `openpyxl`/`pandas` для чтения xlsx,
  `Pillow` для изображений. `requirements.txt` фиксирует версии.
- **Источник данных:** xlsx `Конструктор_офисной_мебели_таблицы_v2.xlsx` (3 листа).
  Копируется в `data/calculator.xlsx`; путь переопределяется в `src/config.py`.
- **Безопасность формул:** без `eval()`. Используем `asteval` (или собственный
  парсер на `ast` с белым списком узлов/имён).
- **Объём расчёта (предположение):** площадь ЛДСП (м²) + длина кромки 0,4 и 2 мм
  (п.м.) + стоимость (₽) по введённым ценам из п.1. Цены не используются впустую.
- **Единицы:** формулы — в мм (мм²); итог площади → м² (÷1 000 000),
  кромка → м (÷1000).

## 3. Реальная структура исходного xlsx (прочитано)
- **`1. Ассортимент`** — столбцы: `Изделие`, `Боковины`, `Топ`, `Дно`,
  `Столешница`, `Полки`, `Перегородки`, `Фасады/двери`, `Ящики`,
  `Задняя стенка`, `Цоколь`, `Опоры`, `Экраны`, `Схема`.
  Числа = кол-во деталей данного типа в изделии. Колонка `Схема` — для ссылки на
  изображение (в v2 пуста → заполняется на этапе подготовки данных).
- **`2. Формулы`** — столбцы: `Изделие`, `Деталь`, `Формула`.
  Примеры: `(В-16)×Г`, `Ш×Г`, `(Ш-32)×Г`, `(Ш-32)×(Г-16)`, `(В-32)×Г`,
  `(Ш-32)×H`, `Ш×Г (2шт)`, `(Ш-32)×500`, спец-токены `по проему`, `инд.`,
  `по модели`, диапазон `300..500`.
- **`3. Кромление`** — столбцы: `Тип детали`, `L1`, `L2`, `S1`, `S2`.
  Значения: `2`, `0.4`, `нет`. Задаёт, какие кромки (длинные L / короткие S)
  и какой толщины кромятся. Пример: Столешница — все 4 грани `2`;
  Боковина — L1,L2 = `0.4`, S1,S2 = `нет`; Полка — только L2 = `0.4`.

## 5. Модель данных и словарь переменных
- Габариты изделия (ввод пользователя, мм): `Ширина`, `Высота`, `Глубина`.
- Переменные в формулах: `Ш`→Ширина, `В`→Высота, `Г`→Глубина, `H`→Высота
  (привести `В` и `H` к одной переменной «высота»; **уточнить у технолога**).
- `Ассортимент` → словарь `{изделие: {тип_детали: количество}}`.
- `Формулы` → словарь `{(изделие, деталь): строка_формулы}`.
- `Кромление` → словарь `{тип_детали: {L1,L2,S1,S2}}`.
- Деталь участвует в площади ЛДСП, только если для неё есть формула.

## 6. Движок формул (`formula_engine.py`)
Каждая формула = произведение **1–2 множителей** (напр. `(В-16)` и `Г`);
это даёт и площадь, и стороны панели для расчёта кромки.
- Подстановка переменных `{Ш,В,Г}` → числа; константы (16, 32, 500) — как есть.
- `×`/разделитель множителей: `*` (нормализуем `×`→`*`).
- Множитель-суффикс `(Nшт)` в конце → умножить площадь на N (напр. `Ш×Г (2шт)`).
- Диапазон `a..b` (напр. `300..500`) → берём середину или параметр с дефолтом;
  вынести в поле ввода, пометить «параметр».
- Спец-токены `по проему`, `инд.`, `по модели` → движок возвращает `None`,
  UI показывает поле ручного ввода площади детали (м²).
- Возвращает `(площадь_мм2, [сторона_a, сторона_b])`.
- Безопасность: `asteval` с разрешёнными именами `{Ш,В,Г}` и арифметикой.

## 7. Расчёт кромки и стоимости (`edge.py`, `calculator.py`)
- Для панели со сторонами `(a,b)`: длинная грань = `max(a,b)`, короткая = `min(a,b)`.
  `L1,L2` → длинные грани, `S1,S2` → короткие (по листу `Кромление`).
- Длина кромки 0,4 мм на 1 деталь = сумма длин граней со значением `0.4`;
  аналогично для `2`. Умножается на кол-во деталей типа.
- Итог: `площадь_ЛДСП_м2 = Σ(кол-во × площадь_мм2) / 1e6`;
  `кромка_0.4_пм`, `кромка_2_пм` (мм→м);
  `стоимость = площадь_м2×цена_ЛДСП + кромка_0.4×цена_0.4 + кромка_2×цена_2`.
- Вывод: сводка + табличка разбивки по деталям (кол-во, площадь, кромка).

## 8. Поток UI (Streamlit, `app.py`)
1. **Блок цен** (sidebar/inputs): цена ЛДСП ₽/м²; цена кромки 0,4 ₽/п.м.;
   цена кромки 2 мм ₽/п.м. (дефолты из `config.py`).
2. **Габариты** (мм): Ширина, Высота, Глубина.
3. **Выбор изделия** — выпадающий список уникальных `Изделие` из `Ассортимент`.
4. По выбору — **визуализация**: изображение из `Схема` → `data/images/<файл>`
   (если пусто — плейсхолдер «нет схемы»).
5. Если у изделия `Полки>0` или `Ящики>0` — **числовые поля** ввода их количества
   (дефолт = значение из `Ассортимент`, редактируемо). При необходимости —
   аналогично `Перегородки`/`Фасады/двери`.
6. **Расчёт** (авто при изменении): площадь ЛДСП (м²), кромка 0,4 и 2 мм (п.м.),
   стоимость (₽) + таблица разбивки. Поля ручного ввода для спец-токенов.

## 9. Роли проекта
1. **Заказчик/Куратор** — требования, утверждение чисел, владелец xlsx.
2. **Технолог-конструктор** — ведёт Ассортимент/Формулы/Кромление, изображения, актуальность формул.
3. **Системный архитектор** — структура приложения, схема данных, контракт формул.
4. **Разработчик ядра (Python)** — loader, движок формул, кромка, калькулятор.
5. **Разработчик UI (Streamlit)** — экраны ввода/выбора/визуализации/вывода.
6. **QA-инженер** — верификация расчётов на эталонных изделиях, тесты.
7. **Техписатель** — README, руководство, спецификация формул.

## 10. Предложение по AI-агентам (кто исполняет какую роль)
- **Architect Agent** → роль 3: проектирует контракт данных и синтаксис формул.
  Выход: `docs/data-contract.md`, `docs/formula-spec.md`.
- **Data/Excel Agent** → роль 2: нормализует xlsx, заполняет `Схема`, раскладывает
  изображения. Выход: `data/calculator.xlsx`, `data/images/`.
- **Formula Engine Agent** → часть роли 4: безопасный вычислитель (×-множители,
  спец-токены, диапазоны). Выход: `src/formula_engine.py` + тесты.
- **Backend/Calc Agent** → роль 4: `src/loader.py`, `src/edge.py`, `src/calculator.py`.
- **UI Agent** → роль 5: `app.py`.
- **QA Agent** → роль 6: эталонные изделия + сверка с ручным расчётом.
  Выход: `tests/`, `out/verification_report.md`.
- **Docs Agent** → роль 7: `README.md`, `docs/user-guide.md`.
- **Curator (человек)** → роль 1: утверждает, поставляет исходный xlsx и изображения,
  проверяет числа на этапе верификации.

## 12. План валидации
- Юнит-тесты движка на формулах из файла: `(В-16)×Г`, `(Ш-32)×(Г-16)`,
  `Ш×Г (2шт)`, `(Ш-32)×500`.
- Интеграционный тест на 3–5 изделиях (Рабочий стол, Шкаф для документов,
  Стойка ресепшн, Бенч-система) со сверкой площади и кромки с ручным расчётом
  технолога (допуск по площади ±1%, по стоимости ±1%).
- Проверка крайних случаев: изделие без полок/ящиков (нет поля ввода),
  формула со спец-токеном (поле ручного ввода), диапазон `300..500` (параметр).

## 13. Открытые вопросы / риски
- **`Схема`**: подтвердить, что колонка содержит имя файла изображения; если нет —
  вести отдельный `images.csv` (изделие→файл).
- **`В` vs `H`**: считать ли одной переменной «высота» (вероятно да).
- **Спец-токены** `по проему`/`инд.`/`по модели`: требуют ручного ввода или
  доп. формул — вне автоматики в v1.
- **Какие детали — ЛДСП:** Опоры/Цоколь/Задняя стенка могут быть не ЛДСП; правило —
  только детали с формулой участвуют в площади (уточнить у технолога).
- **Баш недоступен** в этой среде (sandbox блокирует bash/python); реализующий
  агент проверяет разбор xlsx локально через `openpyxl`.
