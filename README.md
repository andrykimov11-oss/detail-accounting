# Подетальный учёт производства корпусной мебели

Внешнее приложение для учёта прохождения **детали** (а не заказа) по операциям
цеха: распил → кромление → сверление → упаковка.

Учёт в 1С ведётся в разрезе заказа, но физический объект, идущий через участки —
деталь. Заказы неравномерны (1–1000 деталей), поэтому «заказ на участке» не
говорит ни о загрузке, ни о потерях. Это приложение считает факт по деталям и
возвращает в 1С готовый статус операции.

Подробнее: [`docs/architecture.md`](docs/architecture.md).

## Как это работает

```
       план                              факт
  .xbir (Базис-раскрой) ──►  приложение  ◄──── сканер (QR бирки)
                                 │
  план 1С (.xlsx) ──────────────►│
                                 ▼
                    статус операции: «все детали отсканированы?»
                                 │
                                 ▼
                    HTTP POST → сервис 1С → статус установлен
```

Операция получает статус «Выполнено» **автоматически**, когда отсканированы все
плановые детали. Если план 295, а отсканировано 294 — операция не закрывается, а
недостающая деталь попадает в реестр потерь.

## Установка

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Python 3.10+. Ядро учёта работает на стандартной библиотеке; `openpyxl` нужен
только для выгрузок 1С и обезличивания.

## Быстрый старт

```bash
# 1. Создать БД и справочник участков
python main.py --db prod.db init

# 2. Разрешить связку заказов с документами 1С
python main.py --db prod.db link samples/6564/*.xbir \
    --plan "производственные операции.xlsx"
python main.py --db prod.db pending          # что требует технолога

# 3. Загрузить плановый состав деталей (только связанные заказы)
python main.py --db prod.db import samples/6564/*.xbir --require-link

# 4. Обработать скан (в бою это делает интерфейс оператора)
python main.py --db prod.db scan \
    --order 6564 --qr 3c39598e6a \
    --area area_edging --operator op_mazein \
    --op "Облицовывание кромки 19/0,8"

# 5. Правила отбора деталей из справочника цеха
python main.py --db prod.db rules --areas "Операции по участкам.xlsx"

# 6. Сверка план/факт и статус операций
python main.py --db prod.db status --order 6564 \
    --ops "Раскрой плиты 16мм" "Облицовывание кромки 19/0,8"
```

## Структура

```
main.py                    CLI ядра
src/
  pipeline.py              оркестратор: связка → импорт → скан → статус
  one_c_loader.py          выгрузки 1С: заказы, клиенты, справочники
  order_resolver.py        связка «заказ Базис ↔ документ 1С»
  operation_rules.py       правила отбора деталей из справочника 1С
  one_c_writer.py          запись статуса операции в 1С (очередь, HTTP)
  nanxing_parser.py        присадка: разбор .SCX (нормирование участка)
  xbir_parser.py           .xbir (Базис-раскрой) → модель детали
  operation_mapping.py     правила «операция 1С → какие детали в неё входят»
  operation_plan.py        плановый состав операции (шт / м² / пог.м)
  scan_processor.py        событие сканера → факт по детали
  status_calc.py           сверка план/факт, статус операции, payload для 1С
  storage.py               SQLite: детали, события, факты, справочники
  validate_dataset.py      проверка гипотез спецификаций на массиве (Этап C)
  anonymizer.py            обезличивание ПД (ФЗ-152)
tests/                     227 тестов, pytest
docs/                      спецификации и архитектурные решения
```

## Тесты

```bash
python -m pytest          # 227 тестов, ~2 секунды
```

Реальные данные цеха содержат ПД и в репозиторий не попадают (`samples/` в
`.gitignore`). Тесты работают на синтетическом `.xbir`, который собирается в
`tests/conftest.py` по структуре выгрузки Базиса — это даёт воспроизводимость
без наличия конкретного заказа.

Связка заказов покрыта тестами на **реальных спорных заказах** из массива
(`tests/test_order_resolver.py`) — это регрессия на фактических данных цеха.

Часть тестов **фиксирует известные дефекты** как есть (они помечены `ДЕФЕКТ` в
docstring) — чтобы поведение не изменилось незаметно до того, как их починят.
Список: [`docs/known-issues.md`](docs/known-issues.md).

## Валидация данных перед пилотом

Спецификации написаны по анализу одного заказа (47 уникальных деталей). Перед
пилотом гипотезы нужно проверить на полном массиве:

```bash
python src/validate_dataset.py samples/ --out out/ --plan production_plan.xlsx
```

Скрипт проверяет уникальность QR на всём массиве, извлекаемость номера заказа,
покрытие операций правилами, распределение `qty` и долю битых строк.

## Обезличивание данных

Перед передачей данных за пределы контура:

```bash
python src/anonymizer.py samples/ out/anonymized/
```

Замены детерминированы (один клиент → один псевдоним во всех файлах), связи
сохраняются, словарь замен пишется отдельно.
